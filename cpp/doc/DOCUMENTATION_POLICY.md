# 文档维护策略

**版本**: 1.0  
**最后更新**: 2025-12-19

本文档说明项目的文档维护策略和规范。

## 文档语言策略

### 原则

- **中文文档优先**：所有用户文档和开发者文档使用中文
- **代码注释**：使用英文，避免Unicode显示问题，保持与现有代码风格一致
- **API文档**：Doxygen生成，使用英文（Doxygen默认输出）

### 适用范围

- **用户文档**：配置文档、系统架构文档等 → 中文
- **开发者文档**：开发指南、C接口文档等 → 中文
- **代码注释**：头文件、源文件中的注释 → 英文
- **API文档**：Doxygen自动生成的文档 → 英文

## 文档更新流程

### 何时更新文档

1. **功能变更时**：
   - 添加新功能时，更新相关文档
   - 修改现有功能时，更新相关文档
   - 废弃功能时，在文档中标注

2. **配置变更时**：
   - 配置格式变化时，更新配置文档
   - 添加新配置项时，更新配置文档和示例

3. **架构变更时**：
   - 系统架构变化时，更新架构文档
   - API变化时，更新API文档

4. **发现错误时**：
   - 发现文档错误时，及时修正
   - 发现文档不完整时，及时补充

### 文档更新责任人

- **代码作者**：负责更新相关代码的文档
- **功能开发者**：负责更新功能相关文档
- **文档维护者**：负责文档的整体维护和审查

### 文档同步

- 文档应与代码保持同步
- 重大变更时，文档更新应包含在同一个PR中
- 定期检查文档与代码的一致性

## 文档审查

### 审查流程

1. **提交前审查**：
   - 提交者自行检查文档完整性
   - 检查文档格式和链接有效性

2. **PR审查**：
   - 代码审查时同时审查文档
   - 确保文档更新与代码变更一致

3. **定期审查**：
   - 定期检查文档是否过时
   - 检查文档链接是否有效

### 审查标准

- **准确性**：文档内容与代码实现一致
- **完整性**：覆盖所有必要内容
- **可读性**：结构清晰，语言流畅
- **一致性**：术语统一，格式统一
- **可维护性**：易于更新和维护

### 审查清单

- [ ] 文档内容准确，与代码一致
- [ ] 文档结构清晰，易于导航
- [ ] 代码示例可运行
- [ ] 所有链接有效
- [ ] 术语使用一致
- [ ] 格式符合规范

## 文档版本管理

### 版本标识

- 在文档开头添加版本信息和最后更新时间
- 格式：`**版本**: V3` 和 `**最后更新**: YYYY-MM-DD`

### 版本兼容性

- 明确标注文档对应的代码版本
- 重大版本变更时，创建新版本文档或明确标注兼容性

## 文档生成

### API文档生成

- **本地生成**：使用 `doxygen .doxygen-config` 生成
- **CI生成**：将来由CI pipeline自动生成并发布到GitHub Pages
- **生成位置**：`cpp/doc/api/`（或配置指定的目录）
- **注意**：生成的HTML文档可以删除，不需要提交到版本控制

### 文档索引生成

- 文档索引（`doc/README.md`）手动维护
- 添加新文档时，及时更新索引

## 文档模板

参见 `doc/TEMPLATE.md`（如创建）了解文档模板格式。

## 反馈和改进

如发现文档问题或有改进建议，欢迎通过Issue或Pull Request反馈。
